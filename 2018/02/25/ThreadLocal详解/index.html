ThreadLocal详解

发表于 2018-02-25 | 分类于 Java ， 多线程

ThreadLocal说明

ThreadLocal通过增加资源来保证所有对象的线程安全，如我们有100人填写表格，当我们只有一只笔时，那么大写就只能挨个填写，这时候就需要对笔进行加锁。从另一个角度出发，我们准备100支笔，那就不需要对笔进行加锁，可以很快的完成表格的填写。而ThreadLocal就相当于每个人有一支笔。

ThreadLocal提供了线程的局部变量，不是多个线程共有的，ThreadLocal的变量在被多个线程使用时候，每个线程只能拿到该变量的一个独立的副本，各个线程之前的操作没有任何影响。没有线程共享变量的问题

  • 不是解决变量共享的问题
  • 独立的实例副本
  • 没有同步问题，和synchronized没有关系

既然ThreadLocal是Thread的本地变量，因此在Thread退出的时候对ThreadLocalMap的清除：

如果我们在使用线程池时，那就意味着当前线程未必会退出（比如固定大小线程池，线程问题存在）。如果这样，将一些比较大的对象设置到ThreadLocal中，可能会使系统出现内存泄漏，此时如果希望及时回收对象，最好使用ThreadLocal#remove()进行变量移除，另外JDK也可以像释放普通变量一样，比如我们有时候为了加速垃圾回收，可以类似的obj = null之类的代码，加速回收。

JDK8垃圾回收

ThreadLocalMap的实现类似一个HashMap，更精确的说是类型于WeakHashMap。

ThreadLocalMap实现使用了弱引用，弱引用比强引用弱的多的引用，Java虚拟机在垃圾回收时，如果发现是弱引用，就会立刻回收。ThreadLocalMap内部由一系列的Entry构成，每个Entry都是WeakReference

这里的参数K就是Map的K，v就是Map的value，其中k也就是ThreadLocal的实例。

当没有强引用指向 ThreadLocal 变量时，它可被回收，从而避免了 ThreadLocal 不能被回收而造成的内存泄漏的问题。

但是，这里又可能出现另外一种内存泄漏的问题。ThreadLocalMap 维护 ThreadLocal 变量与具体实例的映射，当 ThreadLocal 变量被回收后，该映射的键变为 null，该 Entry 无法被移除。从而使得实例被该 Entry 引用而无法被回收造成内存泄漏。

适用场景：

  • 每个线程需要有自己单独的实例
  • 不希望被多线程共享数据

总结

  • ThreadLocal 并不解决线程间共享数据的问题
  • ThreadLocal 通过隐式的在不同线程内创建独立实例副本避免了实例线程安全的问题
  • 每个线程持有一个 Map 并维护了 ThreadLocal 对象与具体实例的映射，该 Map 由于只被持有它的线程访问，故不存在线程安全以及锁的问题
  • ThreadLocalMap 的 Entry 对 ThreadLocal 的引用为弱引用，避免了 ThreadLocal 对象无法被回收的问题
  • ThreadLocalMap 的 set 方法通过调用 replaceStaleEntry 方法回收键为 null 的 Entry 对象的值（即为具体实例）以及 Entry 对象本身从而防止内存泄漏
  • ThreadLocal 适用于变量在线程间隔离且在方法间共享的场景
