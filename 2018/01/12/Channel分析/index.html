Channel分析

发表于 2018-01-12 | 分类于 Java ， Netty

Netty核心组件

Netty核心组件包含Channel，每一个Channel包含一个ChannelPipeline，而Channel的读写是通过ByteBuf进行操作的，由此可以看到Channel是Netty最核心的组件。

同样在JavaNio中最重要的也是Channel，Channel注册到Selector，而Channel的读写是通过Buffer。

JavaNio的Channel与Netty的Channel对比

Netty重新定义了Channel，和java nio提供的Channel几乎完全不一样，哪Netty为什么自己实现Channel呢？

  • java nio channel和netty channel完全不一样，netty对channel进行了重新实现，java nio channel并没有进行了完全的封装，而netty作为一个通信框架，对channel进行了完全的封装。为SocketChannel和ServerSocketChannel进行了统一的封装，提供了统一的操作视图，而java nio channel没有。
  • JDK的SocketChannel和ServerSocketChannel的主要职责就是网络I/O操作，它们是通过SPI类接口，由具体的虚拟机厂家提供的，所以其实现难度没有太大区别。
  • 而Netty Channel需要和Netty整体框架融合在一起，所以需要对Channel进行重新封装。
  • 自定义Channel更加灵活。

Netty自己实现了Channel。

Netty Channel类图

UnSafe中Channel的一个内部内，UnSafe进行读写操作的。

有两个重要的实现类：

  • NioServerSokcetChannel和NioSocketChannel，这两个类可以类比java nio中的ServerSocketChannel和SocketChannel。
  • NioMessageUnsafe是AbstractNioMessageChannel内部类
  • NioByteUnsafe是AbstractNioByteChannel的内部类
  • AbstractNioUnsafe是AbstractNioChannel的内部类
  • AbstractUnsafe是AbstractChannel的内部类
  • AbstractUnsafe进行register到NioEventLoop上以及bind端口操作
  • AbstractChannel$AbstractUnsafe#register0()进行注册到EventLoop上
  • NioServerSocketChannel#doBind()进行编写端口操作

  • SocketChannel注册到selector

channel对比总结

由上面的描述可以看到Channel通过AbstractUnsafe进行注册操作，由NioServerSocketChannel进行绑定端口操作。完成统一操作视图的功能（好像java nio是通过其它方式进行统一视图的，有时间再分析）。

ByteBuffer

而在NIO编程中，常用的是ByteBuffer

这里分析一下ByteBuffer的缺点：

  1. ByteBuffer不可能动态的扩展和收缩，只能分配固定的 ==长度==。
  2. ByteBuffer只有一个位置指针 position，读写里需要手工调用flip()和rewind()
  3. ByteBuffer的API功能有限

Netty Buffer

netty使用ByteBuf进行数据缓冲区，ByteBuf是一个Byte数组的缓冲区，它的基本功能应该与JDK的ByteBuffer一致。

ByteBuf通过两个位置据指针来协助缓冲区的读写操作，读使用readerIndex，写操作使用writerIndex。

由于写操作不修改readerIndex，而读操作不修改writerIndex，因此读写之前不再需要调整位置指针，这样就极大的简化了缓冲区的读写操作，避免了由于遗漏或者不熟悉flip()操作导致的功能异常。
